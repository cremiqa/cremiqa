# Contributing to Cremiqa

First of all, thank you for your interest in contributing to Cremiqa.

Our goal is to build a modular, machine-independent coffee automation platform with a strong emphasis on clarity, maintainability and long-term evolution.

## Start Here

Read the following documents in order:

1. README.md
2. docs/glossary.md
3. Accepted Architecture Decision Records (docs/adr/)

The glossary defines the platform vocabulary.

The ADRs explain why architectural decisions were made.

## Engineering Principles

- Prefer readability over cleverness.
- Keep solutions simple.
- Respect the established architectural boundaries.
- Avoid implementation-specific assumptions in architectural discussions.
- Favor composition over specialization where appropriate.
- Preserve the semantic meaning of existing concepts.

## Architecture Changes

Architecture evolves through discussion before implementation.

If your contribution changes the architecture:

- Discuss the proposal first.
- Create or update an ADR when appropriate.
- Update the glossary if architectural terminology changes.

## Pull Requests

Please keep pull requests focused.

A good pull request should:

- solve one problem,
- include documentation updates where appropriate,
- remain consistent with accepted ADRs,
- avoid unrelated refactoring.

## Documentation

Documentation is considered part of the project.

Whenever necessary, update:

- ADRs
- Glossary
- README
- Developer documentation

Documentation should evolve together with the implementation.

## Communication

Constructive discussion is encouraged.

Challenge ideas rather than people.

When in doubt, prefer asking questions over making assumptions.
